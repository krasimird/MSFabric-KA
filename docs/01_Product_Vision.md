## 1. Проблемът

Екипът за разработка на GenDWH (Generali Data Warehouse) ежедневно се сблъсква с едни и същи въпроси: какви полета има тази таблица, откъде идва стойността на това поле, как се изчислява cat_id, какво означава UoA, защо pipeline-ът е структуриран по този начин.

Отговорите на тези въпроси са разпръснати в няколко места:

- **108+ SQL заявки** в metadata таблици на Administration Lakehouse, голяма част с многослойни CTE вериги които са трудни за четене
- **Pipeline JSON definitions** с десетки activities и dependency chains
- **Устно знание** в главите на 2-3 души от екипа
- **Статични Word/Excel документи** които остаряват в деня в който са създадени

Резултатът е загуба на време, повтарящи се обяснения, и риск от грешки когато колега приеме нещо за вярно без да провери.

## 2. Визията

**GenDWH Knowledge Assistant** е система която прави знанието за платформата достъпно, актуално и разбираемо. Тя се състои от два компонента:

### 2.1 Автоматична документация (Documentation Engine)

Fabric Notebook който се изпълнява по schedule и:

- **Открива** всички workspaces и items в Fabric tenant-а чрез REST API
- **Извлича** metadata — pipeline definitions, table schemas, SQL заявки
- **Анализира** SQL заявките с AI (Claude API) — резолвира CTE вериги, идентифицира трансформации, описва бизнес логиката
- **Генерира** документация в Word и Excel формат с InspirIT брандинг
- **Версионира** — записва какво се е променило от последния run

Документацията не е еднократна — тя се обновява автоматично при всяка промяна в платформата.

### 2.2 Q&A интерфейс (Knowledge Assistant)

Web приложение достъпно от браузър, което позволява на всеки от екипа да зададе въпрос на естествен език и да получи отговор с конкретни данни от платформата:

- "Какви полета има dim_policy?"
- "Откъде идва полето suma в stg_claim_expenses?"
- "Как се изчислява cat_id?"
- "Обясни ми pipeline GenDWH_Orchestration_DP"
- "Кои таблици захранват fact_premiums_written?"

Отговорите не са общи — те са базирани на реалните metadata, SQL изрази и lineage данни от платформата.

## 3. Целева аудитория

| Роля | Как използва системата |
|------|----------------------|
| **Data Engineer** | Пита за lineage, трансформации, pipeline dependencies при разработка |
| **BI Developer** | Пита за полета, типове данни и бизнес значение при създаване на отчети |
| **Business Analyst** | Пита за бизнес термини, IFRS 17 концепции, какво означава дадено поле |
| **QA / Тестер** | Проверява очаквани трансформации, source tables, join keys |
| **Нов колега** | Onboarding — разбира архитектурата и потоците без да пита колеги |
| **Мениджмънт** | Получава актуална документация без да чака ръчно писане |

## 4. Ключови принципи

### Документацията е код
Документацията се генерира от metadata, не се пише ръчно. Когато SQL заявка се промени, документацията се обновява автоматично на следващия scheduled run.

### Знанието е достъпно, не скрито
Не трябва да знаеш къде да търсиш или как да четеш SQL за да разбереш как работи полето `suma`. Задаваш въпрос — получаваш отговор.

### Няма single point of failure
Ако утре единственият човек който знае IFRS логиката напусне, знанието остава в системата — извлечено, анализирано и обяснено.

### Environment-aware
Системата разбира naming conventions (`_WS_D`, `_WS_T`, `_WS_P`) и документира правилния environment. Утре ако се появи нов workspace — автоматично влиза в обхвата.

## 5. Какво НЕ е продуктът

- **Не е ETL tool** — не изпълнява и не променя pipeline-и
- **Не е data catalog** (като Microsoft Purview) — не управлява data governance policies
- **Не е BI tool** — не визуализира данни и не създава отчети
- **Не замества хората** — дава им информация за да вземат по-бързи и по-правилни решения

## 6. Мерки за успех

| Метрика | Текущо | Цел |
|---------|--------|-----|
| Време за отговор на "какви полета има таблица X" | 5-15 мин (ровене в SQL) | < 10 сек |
| Време за onboarding на нов колега | 2-3 седмици | 3-5 дни |
| Актуалност на документацията | Месеци изоставане | < 24 часа |
| Брой повтарящи се въпроси към senior колеги | Няколко на ден | Близо до нула |

## 7. Ограничения и рискове

| Риск | Митигация |
|------|----------|
| Claude API downtime | Кеширане на предходни отговори; документацията е статична и не зависи от uptime |
| Некоректен AI анализ на сложен SQL | Validation чрез сравнение с ръчен анализ на 10-15 ключови таблици; feedback loop |
| API cost при голям обем въпроси | Sonnet е ~$0.003/въпрос; при 100 въпроса/ден = $6.60/месец |
| Промяна в Fabric REST API | Notebook-ът е модулен; API extraction е изолирано в отделна фаза |
| Сигурност на API ключа | Всеки потребител въвежда собствен ключ; никъде не се пази на сървъра |
